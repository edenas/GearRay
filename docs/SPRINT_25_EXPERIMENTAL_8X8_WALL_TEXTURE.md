# Sprint 25 — Experimental Native 8×8 Wall Texture

## Purpose and status

This isolated experiment tests whether a genuine native 8×8 wall texture is a
better final fit for GearRay than the Phase 1 16×16 baseline. The implementation
and deterministic validation are complete, but the visual acceptance decision
is intentionally pending Emulicious and original Game Gear comparison. The
experiment remains uncommitted and reversible from `gearray-engine-v1.0`.

The supplied `wall_concrete.png` is the active source of truth. It is an 8×8,
24-bit RGB PNG with SHA-256
`F33A21A9F871A407E21887C2FD6C0EAC862F07BA99A22987D7BEF8F1282A6E03`.
No resizing, filtering, blurring, or artwork regeneration is performed.

## Native packed format

- Source pixels: 64 (8 columns × 8 rows)
- Indexed colour range: 0–6
- Packing: two 4-bit indices per byte, left pixel in the high nibble
- Row stride: 4 bytes
- Rows: 8
- Generated texture data: 32 bytes
- Storage: constant ROM data; persistent sampler RAM is unchanged

The generated bytes were independently reconstructed from the RGB source,
generated palette, nearest-colour indexing, and high/low-nibble packing. All 32
bytes matched, and each row contains exactly four bytes.

## Horizontal mapping and addressing

The oriented ray hit offset remains 0–255. Native texture X is
`hit_offset >> 5`, so every source column receives exactly 32 consecutive hit
offsets. Packed byte X is `hit_offset >> 6`. Hit-offset bit 5 selects the nibble:
bit 5 clear selects the high nibble and bit 5 set selects the low nibble.

All 256 hit offsets were exhaustively checked against texture X 0–7, packed byte
0–3, and source-X parity. The sampler retains its compact combined byte/nibble
field; its mask changes from `0x07` to `0x03`.

## Vertical mapping

For projected height `h`, the generator records:

```text
step      = floor((8 × 256) / h)
remainder = (8 × 256) - step × h
```

The sampler advances the Q8.8 position and carries the remainder exactly as in
the Phase 1 implementation. The generated release tables cover every reachable
textured height from 17 through the engine projection cap of 144. An exhaustive
model check covered heights 1–255, verified the exact source row
`floor(screen_pixel × 8 / h)`, proved that all eight rows are reachable for
heights of at least eight, and exercised unclipped and top/bottom-clipped walls.
Heights 145–255 are validated mathematically but remain unreachable under the
existing 144-pixel projection cap.

## LOD, palette, and renderer interaction

LOD behavior is unchanged:

- height greater than 16: native textured sampling
- height 9–16: existing flat distant-wall colour
- height 1–8: existing very-distant fog colour

Only the near path initializes and reads the sampler. X-side indices still add
palette base 2 and Y-side indices add palette base 9. Both sides and all seven
source indices remain within the established 0–15 native palette range. The
LOD thresholds, silhouette rules, dirty signature, tile packing tables, VRAM
upload calls/order, viewport border, raycaster, DDA, projection, banking, and
gameplay code were not changed.

## Measurements

| Measurement | 16×16 baseline | 8×8 experiment | Difference |
| --- | ---: | ---: | ---: |
| Source dimensions | 16×16 | 8×8 | native replacement |
| Packed texture data | 128 bytes | 32 bytes | -96 bytes |
| Packed row stride | 8 bytes | 4 bytes | -4 bytes |
| Release ROM used | 14,029 bytes | 13,931 bytes | -98 bytes |
| Profiling ROM used | 15,077 bytes | 14,979 bytes | -98 bytes |
| Bank 0 used | 13,252 bytes | 13,154 bytes | -98 bytes |
| Bank 0 free | 3,132 bytes | 3,230 bytes | +98 bytes |
| Bank 1 used/free | 777 / 15,607 bytes | 777 / 15,607 bytes | unchanged |
| Persistent RAM | 934 bytes | 934 bytes | unchanged |
| Sampler state | 15 bytes | 15 bytes | unchanged |
| Sampler init instructions | 105 | 104 | -1 emitted instruction |
| Sample-next instructions | 116 | 115 | -1 emitted instruction |
| Step/remainder tables | 384 bytes | 384 bytes | unchanged |

The two sampler functions shrink from 221 to 219 emitted instructions and by
two code bytes in total. Shift counts, masks, row stride, and table contents are
simpler; function calls, stack use, and renderer call structure do not change.
Main, `raycaster_update`, `cast_ray`, renderer draw, and sampler-initialize stack
frames remain 1, 19, 35, 57, and 10 bytes respectively. No FPS improvement is
claimed without runtime measurement. Native tile output and therefore VRAM
bytes/calls are unchanged.

## Visual comparison checklist

The following items require manual Emulicious and original-hardware testing and
are not marked complete by deterministic validation:

- front-facing, angled, X-facing, and Y-facing walls
- very close and medium-distance textured walls
- LOD 1 and LOD 2 walls
- source orientation and high/low-nibble order
- skipped rows/columns or periodic corruption
- close-range repetition and perceived texture fidelity
- movement, rotation, and simultaneous movement/rotation

An 8×8 texture necessarily repeats or enlarges detail more aggressively than a
16×16 source on close walls. The supplied artwork also quantizes mostly to one
neutral indexed colour, so subjective detail and material readability are the
central hardware-test questions rather than correctness or ROM size.

## Recommendation

Retain the 8×8 implementation in the working tree for the requested manual A/B
test, but do not declare it the permanent engine format yet. Adopt it only if
Emulicious and original Game Gear testing find the supplied artwork at least as
readable and less objectionably repetitive than the tagged 16×16 baseline.
Otherwise restore the complete implementation and asset from
`gearray-engine-v1.0`; the measured 98-byte ROM saving alone is not sufficient
justification.
